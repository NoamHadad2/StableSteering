StableSteering: A Traceable Platform for Interactive Preference-Guided Diffusion Steering

Abstract

StableSteering is a traceable platform intended to support research on prompt-first, preference-guided multi-round diffusion refinement. Starting from a user text prompt, the system generates candidate images, records explicit user preferences, updates a steering state z, and preserves replay, diagnostics, and trace artifacts for post-hoc inspection. The implementation includes a FastAPI backend, GPU-backed Diffusers inference, per-session YAML configuration, SQLite persistence, and browser-based interaction flows. The repository also includes a checked-in qualitative multi-round case study, an executed repeated-run pilot baseline matrix, and local backend and browser verification. The present paper reports a traceable platform together with two bounded artifact-based evaluations; it does not claim algorithmic superiority or a human-subjects study.

1. Introduction

Text-to-image diffusion systems are powerful, but practical control remains awkward during local refinement. A user may be able to recognize which of several images is better, yet still be unable to express the corresponding improvement as a concise prompt rewrite. The motivation documents in this repository frame StableSteering around that gap: instead of treating prompt editing as the only control surface, the system treats iterative user preference as the main supervisory signal and organizes generation as a repeated loop of prompt, candidate proposal, judgment, update, and replay (docs/motivation.md).

StableSteering addresses that gap with a replayable preference-guided loop. The system starts from a user prompt, generates multiple candidate images, records structured feedback, updates a steering state z, and produces the next round. In the current implementation, z is a compact session-level control state updated from normalized preferences and used to condition subsequent generation; it should be read as an engineering choice rather than a theoretically validated latent representation. The repository already contains a runnable application, a real GPU-backed Diffusers backend, replay export, diagnostics, backend and frontend tracing, configurable samplers and updaters, and a preserved real qualitative case study (README.md; app/main.py).

The defensible contribution is a platform contribution: StableSteering contributes a runnable, GPU-backed, replayable environment for iterative image steering, and this paper demonstrates the workflow and artifacts rather than claiming benchmark superiority or a best steering algorithm. The evidence in the paper comes from software verification, one curated qualitative trace, and one bounded workflow-comparison pilot; no human-subjects study is reported. After this introduction, the manuscript first positions the work relative to nearby diffusion-control literature, then explains the platform and steering loop, then presents the qualitative case study, pilot evidence, verification boundary, limitations, and next evidence threshold.

2. Related Work and Positioning

Prior diffusion-control work differs mainly in what users can control, when that control is applied, and whether feedback updates the model itself or only the session state. Latent Diffusion Models established the practical image-generation backbone that StableSteering builds on, while classifier-free guidance became a standard inference-time control mechanism [@rombach2022ldm; @ho2022cfg]. The platform contribution here is therefore not a new base generator, but a platform that organizes user interaction and artifact capture around an existing diffusion backend.

One comparison axis is the user-facing control surface. Prompt-to-Prompt, InstructPix2Pix, Imagic, and SDEdit illustrate prompt-centric, instruction-centric, or edit-guided diffusion control [@hertz2022prompttoprompt; @brooks2023instructpix2pix; @kawar2023imagic; @meng2022sdedit]. These methods differ in how they let the user steer outputs, but they generally keep prompt or edit instructions as the primary interaction primitive. By contrast, StableSteering centers a persistent session with explicit rounds, candidate comparison, structured preference capture, and replayable multi-round history.

A second comparison axis is where intervention occurs in the generation stack. DiffusionCLIP, Imagic, and ControlNet show that diffusion outputs can be manipulated through internal representations, learned controls, or auxiliary conditioning pathways [@kim2022diffusionclip; @kawar2023imagic; @zhang2023controlnet]. StableSteering does not claim a new control primitive of that kind. By contrast, StableSteering changes the interaction state and candidate-selection policy across rounds while relying on an existing diffusion backend for rendering.

A third comparison axis is how preference information is used. Preference-alignment work for text-to-image diffusion, such as DPOK and Curriculum DPO, uses preference or reward signals to fine-tune the generative model itself [@fan2024dpok; @croitoru2025curriculumdpo]. StableSteering’s present contribution is narrower and more infrastructural: it maintains a session-level steering state, updates that state from normalized user feedback, and preserves the resulting interaction trace for replay and audit. By contrast, these preference-optimization methods update model parameters or training objectives, whereas StableSteering updates only the session loop.

Finally, the most relevant systems-oriented comparison axis for this paper is artifact preservation. DiffusionDB matters here because it exposes how users interact with text-to-image systems in practice, including prompt diversity and parameter variation [@wang2023diffusiondb]. StableSteering is motivated by the same practical difficulty of prompt-only control, but its defensible contribution is traceability: replay views, diagnostics, trace bundles, and preserved per-session artifacts make the steering loop inspectable after the fact. The paper therefore keeps its novelty claim at the systems-platform level and avoids “first” language or algorithm-superiority claims until broader literature coverage and more normalized comparisons are added (see paper/related_work_draft.md and paper/references.bib).

3. StableSteering Platform and Core Loop

StableSteering is implemented as a web application with several cooperating layers: a prompt-first frontend, an experiment and session orchestrator, a generation engine, interchangeable sampling and update modules, SQLite-backed persistence, and tracing/reporting utilities. These layers matter because they all support one end-to-end loop: prompt, candidate generation, preference capture, state update, and replayable inspection. The system specification and architecture diagrams in the repository describe these layers in detail (docs/system_specification.md; docs/assets/illustrations/runtime_flow.svg).

At a user-facing level, the workflow is straightforward. The user enters a prompt, optionally edits a per-session YAML configuration, creates a session, generates a round of candidates, submits explicit feedback, and inspects replay or trace outputs. At a systems level, the frontend passes the prompt and YAML to the backend, the backend materializes an experiment/session snapshot, the orchestrator advances the round lifecycle, the generation engine renders candidate images, and the trace layer records both backend and frontend events for later audit (app/core/config_yaml.py; app/engine/orchestrator.py; app/core/tracing.py).

The core session state is defined in the repository schema layer. A session stores the prompt, model name, immutable configuration snapshot, current steering state, round index, and incumbent candidate identifier (app/core/schema.py). In the current implementation, the steering state z is a low-dimensional session-level steering state used to represent the current point in steering space and influence subsequent generation through the backend conditioning path. After each round, the state is updated from normalized user feedback through the configured updater. The paper should treat z as a practical session-level state used by the backend, not as a theoretically optimal representation.

Operationally, the current loop can be summarized as follows:

initialize session with prompt p, config c, and steering state z_0 = 0
for round t = 1, 2, ...
  sample candidate offsets around z_t according to sampler(c)
  render candidates using prompt p, steering state z_t, and generation config c
  collect structured user feedback f_t over rendered candidates
  normalize f_t into updater-facing form
  update steering state z_(t+1) = U(z_t, f_t, c)
  preserve round, feedback, replay, and trace artifacts

In the present implementation, z has configurable dimensionality, is initialized as the zero vector for the session, is updated only from session feedback rather than model training, and affects generation through the generation engine’s steering path and prompt-conditioning logic. This is enough to define the operational role of z in the system even though the paper does not claim that this state is theoretically optimal.

The orchestrator advances the loop one round at a time. On the first round, the system includes an explicit baseline candidate corresponding to the unmodified prompt. In later rounds, the previous winning candidate is carried forward as an incumbent. Additional candidates are proposed by the configured sampler, rendered by the generation engine, and presented for user feedback. Feedback is normalized and then consumed by the configured updater, which computes the next incumbent state (app/engine/orchestrator.py; app/feedback/normalization.py; docs/assets/illustrations/session_lifecycle.svg).

The feedback path is equally important to the loop. The repository supports several feedback forms, including scalar ratings, winner-only choices, pairwise comparisons, approve/reject signals, and top-k style feedback, which are normalized into a common updater-facing structure before the configured updater computes the next steering state (app/feedback/normalization.py; docs/assets/illustrations/feedback_modes.svg). The repository also implements several sampler families and update policies. The safest claim here is modularity: the platform implements interchangeable modules under one lifecycle. These strategies are implemented and lifecycle-tested, not yet comparatively evaluated as better or worse steering algorithms.

4. Implementation and Observability

The backend is built with FastAPI, persistent state is stored in SQLite plus artifact and trace directories, asynchronous jobs expose progress updates, and diagnostics reveal the resolved backend and device (app/main.py; app/storage/repository.py; app/core/jobs.py). These implementation choices matter because they make the platform runnable, inspectable, and auditable rather than leaving it at the level of an abstract interaction concept.

The generation engine supports a real Diffusers backend on GPU by default and treats the mock backend as test-only (app/engine/generation.py). The setup surface also exposes per-session YAML configuration that changes operational settings such as sampler choice, seed policy, steering dimension, trust radius, guidance scale, inference steps, and model name (docs/configuration_manual.md).

StableSteering also preserves session replay, per-session trace bundles, diagnostics views, and HTML reports. This does not by itself prove scientific novelty, but it does make the replayable steering loop concrete and inspectable as a platform artifact (app/core/tracing.py; app/frontend/templates/diagnostics.html).

5. Qualitative Case Study: One Real End-to-End Session

The repository includes a checked-in qualitative case study generated by scripts/create_real_e2e_example.py. The current manifest records a five-round session with twenty-five candidate presentations, real GPU-backed Diffusers inference, and zero failing image-sanity checks (output/examples/real_e2e_example_run/manifest.json). The bundle also includes a standalone HTML walkthrough and a trace report (output/examples/real_e2e_example_run/real_e2e_example_run.html; output/examples/real_e2e_example_run/session_trace_report.html).

This single curated qualitative case study is the paper’s central demonstration of what the platform enables. It shows a replayable steering loop in practice: prompt-first setup, explicit multi-round candidate generation, structured user feedback, carry-forward of the incumbent result, and a preserved artifact bundle that can be inspected afterward. It should not be framed as a benchmark result or proof of general superiority. It is also separate from the executed pilot baseline matrix discussed later in this paper.

The case-study setup should be stated compactly and explicitly. The checked-in run uses the prompt “A premium cinematic product hero photo of an expedition-ready electric explorer motorcycle...” with a negative prompt suppressing blur, flat lighting, geometry distortions, text, and clutter; it uses five candidates per round, random_local sampling, winner_average updating, scalar_rating feedback, 512x512 outputs, and runwayml/stable-diffusion-v1-5 on the Diffusers backend with 30 inference steps. The scripted stopping rule is five rounds. These details matter because they make the qualitative demonstration reproducible as a repository artifact rather than a vague showcase.

6. Verification and Reproducibility

The repository includes substantial local verification for backend behavior and browser flows. During one single-environment verification snapshot while preparing this paper, the backend and browser suites completed successfully, including lifecycle behavior, generation-policy enforcement, tracing, feedback normalization, YAML configuration, and browser interaction flows (tests/test_session_lifecycle.py; tests/test_generation_engine.py; tests/test_feedback.py; tests/e2e/app.spec.js). That snapshot recorded 64 passed backend tests and 8 passed, 1 skipped browser tests.

It is important to separate software verification from research validation. The tests support claims about system functionality, runtime policy, and artifact integrity. They do not, by themselves, establish research conclusions about user benefit or algorithmic superiority. The reproducibility value of the repository lies instead in its preserved session artifacts, explicit runtime policy, configurable sessions, replayable traces, and a concrete checked-in case-study bundle.

The repository now also includes an executed minimal baseline matrix bundle under paper/results/baseline_matrix, produced by scripts/run_paper_minimal_baseline_matrix.py. That pilot comparison runs three illustrative prompts across three bounded policies with three independent repeats per prompt-policy cell and writes per-run summaries, trace reports, and CSV tables. A derived analysis layer under paper/results/baseline_matrix/analysis, produced by scripts/build_paper_baseline_analysis.py, adds normalized workflow summaries without making outcome-quality claims. This improves the paper package’s empirical readiness, but it should still be read as a small pilot comparison rather than a complete benchmark.

The repository now also includes two follow-on controlled bundles that extend this empirical scaffold without broadening the claim boundary. The first, paper/results/seed_policy_slice, compares fixed-per-round and fixed-per-candidate seed policies under matched budgets, a fixed sampler, and a fixed updater. The second, paper/results/updater_ablation, compares three updater choices under a fixed sampler and seed policy. Both bundles include repeated runs, per-run trace reports, and derived analysis summaries; both remain descriptive and workflow-focused.

Minimal Baseline Pilot

To move beyond a single qualitative case study, we executed a bounded minimal baseline matrix over three illustrative prompts and three policies: a prompt-only one-round proxy, no-update random sampling, and the StableSteering default steering loop. The current bundle includes three independent repeats per prompt-policy cell, for 27 runs, 36 rounds, and 144 candidate images on the real Diffusers backend. Because the policies are intentionally different in control structure, this bundle should be read as a workflow-comparison pilot with unequal interaction budgets rather than as a normalized benchmark of outcome quality. The StableSteering default policy is the only policy in the bundle that performs a feedback update and a follow-up round. At the workflow level, the repeated runs show identical workflow counts in this tiny pilot: each prompt-only or no-update run remains a one-round, zero-feedback run, and each StableSteering default run remains a two-round, one-feedback run, with zero observed standard deviation for those counts across the three repeats in every prompt-policy cell. The new analysis layer adds normalized per-run and per-round workflow summaries, again without making image-quality or superiority claims. The run summaries, compact workflow table, repeatability note, appendix-style analysis summary, and per-run trace reports are archived under paper/results/baseline_matrix, including pilot_summary.md, pilot_table.md, seed_robustness.md, and analysis_summary.md. The lightweight visual flags in that bundle should be treated as appendix-style artifact screening only, not as paper-facing quality evidence. This pass does not test image-quality stability or seed robustness under matched budgets.

Controlled Follow-on Slices

To make the paper package more publishable without drifting into a large benchmark campaign, we executed two additional bounded bundles. The budget-normalized seed-policy slice compares fixed-per-round and fixed-per-candidate under the same sampler, updater, prompt subset, and repeat count; it completed 18 runs, 36 rounds, and 180 candidate rows. The fixed-sampler updater ablation compares winner_copy, winner_average, and linear_preference under the same sampler, seed policy, prompt subset, and repeat count; it completed 27 runs, 54 rounds, and 270 candidate rows. In both bundles, the observed workflow counts remain invariant: all runs stay at two rounds and one feedback event under the configured steering-loop budget. These bundles therefore strengthen the repository’s value as a controlled experimentation platform, but they still do not establish image-quality superiority or outcome sensitivity across policies. The derived summaries are preserved under paper/results/seed_policy_slice/analysis and paper/results/updater_ablation/analysis.

7. Discussion and Limitations

The most important limitation is the absence of broad aggregate comparative evidence. The repository now includes an executed repeated-run minimal baseline matrix bundle with three illustrative prompts, three bounded policies, and three independent repeats per prompt-policy cell, plus a small derived analysis layer, but it still does not include a broader prompt suite or a normalized apples-to-apples benchmark comparison across equalized policies and budgets. It also does not yet include an analysis notebook that turns session corpora into paper-ready figures and tables (docs/research_improvement_roadmap.md). The checked-in qualitative run should therefore still be read as a single curated case study rather than a corpus-level evaluation.

There are also narrower interpretation risks. First, the current paper cannot yet determine when the steering loop helps more than prompt-only iteration or no-update sampling; apparent improvement in the qualitative case study could still reflect additional candidate exposure or example selection rather than a general steering advantage. Second, the pilot matrix is intentionally policy-asymmetric: prompt-only and no-update are bounded control policies, while the StableSteering default loop performs a feedback update and a follow-up round, so the pilot should be read as a workflow comparison rather than a normalized benchmark of outcome quality. Third, the new three-repeat extension and derived analysis layer only establish narrow workflow-count repeatability in this tiny matrix; they do not justify significance language, broad robustness claims, or claims about human-perceived image quality. Fourth, the current manuscript still does not characterize sensitivity across broader prompt classes, feedback modes, and equalized policy budgets, so it does not establish that the steering state remains stable when preference signals are noisy or ambiguous. Fifth, the platform is reproducible as software, but not yet fully validated as a research instrument across environments: it depends on a real GPU-backed Diffusers backend, correct runtime configuration, and trace integrity, while the current tests validate system behavior rather than research effectiveness.

Before any new empirical results are interpreted causally, the paper should hold fixed the model/backend, prompt suite, stopping rule, feedback mode, and seed policy, and then add a budget-normalized seed-policy slice plus a fixed-sampler updater ablation. Until then, the manuscript should remain explicitly qualitative in its claims and descriptive in its pilot interpretation.

8. Conclusion and Future Work

StableSteering contributes a traceable, runnable platform for prompt-first, preference-guided multi-round diffusion steering with replay, diagnostics, and preserved session artifacts. The current repository supports that platform claim with a real GPU-backed implementation, modular lifecycle components, a checked-in qualitative case study, and substantial local software verification. It does not yet support a benchmark claim that the steering loop is superior to prompt-only or no-update alternatives.

The next evidence threshold is clear. A stronger paper would build on the executed repeated-run pilot baseline matrix, the completed seed-policy slice, and the completed updater ablation with a broader prompt suite and a reusable analysis-ready export path. The current bundles and derived analyses improve empirical readiness and now document a narrow form of workflow-level repeatability across several controlled settings, but they still do not establish superiority, significance, or generalization. The repository’s roadmap documents already identify compatible next steps in stronger baselines, richer elicitation modes, structured analysis exports, broader diffusion pipeline support, and more formal user-study instrumentation (docs/system_improvement_roadmap.md; docs/research_improvement_roadmap.md).
